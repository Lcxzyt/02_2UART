# 蓝牙串口接反与虚电压反灌问题排查总结

本文记录本次 MSPM0G3507 循迹小车调试过程中遇到的“蓝牙一插系统卡死、断电后蓝牙不彻底重启、5V 轨出现虚电压”等异常现象、排查过程、最终结论和后续建议。

## 1. 背景

当前小车主控为嘉立创天猛星 MSPM0G3507 开发板。系统中同时存在：

- UART0：作为主要调试串口；
- UART1：连接蓝牙串口模块；
- I2C：连接 MPU6050 / QMC5883L 等 IMU 器件；
- 新增感为科技八路灰度 ADC 采样模块。

在实现灰度 ADC 采样和打印后，调试中出现了蓝牙相关的异常。最初怀疑过 ADC/I2C/UART 软件冲突，但最终定位为硬件接线错误和电源反灌问题。

## 2. 主要异常现象

### 2.1 蓝牙插上后系统像“卡死”

现象：

- 蓝牙模块接到开发板 UART1 后，即使蓝牙没有正常上电，系统也会变得像死机；
- OLED 显示不刷新或响应很慢；
- 按 restart 键看起来也不重启；
- 拔掉蓝牙线后，从 OLED 上能看到系统重新响应 restart 操作。

### 2.2 UART0 出现乱码和反复磁力计校准日志

UART0 调试串口出现类似日志：

```text
A-o���MCal CANCEL keep old
MCal START 10s TL=10 TR=-10 rotate in place
MCal CANCEL keep old
MCal START 10s TL=10 TR=-10 rotate in place
MCal CANCEL keep old
...
MCal 10% MX=338 MY=279 MZ=504 MinX=338 MaxX=348 MinY=276 MaxY=290
...
MCal 100% MX=339 MY=284 MZ=507 MinX=333 MaxX=352 MinY=275 MaxY=291
```

其中 `A-o���` 是乱码，后面的 `MCal START / CANCEL` 表明系统不断收到了类似 `M` 的命令字符。

### 2.3 I2C 曾被误判为异常

之前观察到开发板读不到 I2C 数据，甚至 UART0 指令也没有回传，因此一度怀疑：

- ADC 采样改动影响 I2C；
- UART0 和蓝牙 UART1 有资源冲突；
- 某个软件函数阻塞。

后续安全版测试中，`HMpu=1`、`HMagOk=1` 等状态显示 I2C 本身可正常工作，真正问题在蓝牙 UART1 侧。

### 2.4 断开供电后仍有 1.7V 到 2.4V 左右的“虚电压”

用户观察：

- 开发板 5V 没有主动给外设供电时，某些情况下仍能测到接近 2.4V 的电压；
- 空测时后来测不到该电压；
- 接 1k 负载也测不到；
- 接蓝牙模块时可测到约 1.77V；
- 接 CH340 模块时可测到约 2.44V；
- 蓝牙以前偶发“不重启”，必须重启根路电源后才恢复。

这些现象说明系统中存在弱反灌/虚电压路径。

## 3. 排查过程

### 3.1 软件隔离：屏蔽蓝牙 UART1

先做了一个 UART0-only 安全测试版：

- `Bluetooth.c` 中设置：

```c
#define BLUETOOTH_UART_ENABLE 0U
```

- 不启用 UART1 RX 中断；
- 不处理蓝牙 RX；
- 不向蓝牙 TX 发送；
- 蓝牙相关计数 `BTRX / BTIRQ` 应保持为 0；
- `M` 磁力计自动校准命令只允许 UART0 触发，蓝牙输入不允许触发。

测试结果：

- 蓝牙插上并上电后，系统不再被影响；
- UART0 正常；
- I2C 状态正常。

说明主循环、I2C、ADC、UART0 基本没问题，异常集中在蓝牙 UART1 或蓝牙硬件接入。

### 3.2 回滚到蓝牙问题版本复现

之后为了验证问题，把代码回到蓝牙启用版本：

```c
#define BLUETOOTH_UART_ENABLE 1U
```

同时恢复蓝牙也能触发命令的状态。

测试结果：

- 问题复现；
- UART0 出现乱码；
- 反复出现 `MCal START / CANCEL`；
- 蓝牙线拔掉后系统恢复响应。

这说明蓝牙 UART1 RX 接收到了大量异常字节，并被命令解析器当作有效命令处理。

### 3.3 初步判断：不是“读蓝牙阻塞”，而是“乱码 + 中断风暴 + 命令误触发”

蓝牙接收逻辑并不是一直等待数据的阻塞式读取，而是类似：

```c
while (DL_UART_Main_isRXFIFOEmpty(BT_UART_INST) == false) {
    Bluetooth_RingBuf_Put(DL_UART_Main_receiveData(BT_UART_INST));
}
```

也就是说，代码只在 RX FIFO 有数据时读取，空了就退出。它不是：

```c
while (!收到蓝牙数据) {
    一直等;
}
```

因此现象更符合下面链路：

```text
蓝牙 TX/RX 异常或悬空/接错
        ↓
UART1 RX 收到大量乱码
        ↓
UART1 RX 中断高频触发
        ↓
Bluetooth_RingBuf 进入随机字节
        ↓
CmdDispatch 把随机字节当作命令
        ↓
乱码中出现 'M'
        ↓
反复触发 MCal START / CANCEL
        ↓
主循环被中断和命令处理严重干扰，看起来像卡死
```

### 3.4 硬件验证

做过的验证包括：

1. 空测/加 1k 负载测试 5V 虚电压；
2. 只接蓝牙 GND + VCC，不接 TX/RX；
3. 进一步检查蓝牙 TX/RX/VCC/GND 接线。

关键发现：

> 蓝牙模块接反了。

接线纠正后：

- 蓝牙和系统正常工作；
- 不再出现插蓝牙导致系统卡死；
- UART0 不再出现反复 `MCal START / CANCEL` 的异常日志。

## 4. 最终结论

### 4.1 主故障：蓝牙模块接反导致 UART1 异常

本次最主要的问题是蓝牙模块接线错误。接反后可能导致：

- 蓝牙没有正常上电；
- 模块处于半供电/异常状态；
- TXD/RXD 线电平不确定；
- 开发板 UART1 RX 收到大量乱码；
- 单字符命令被误触发；
- 系统表现为卡死或不响应。

### 4.2 UART0 日志中的 `MCal START / CANCEL` 是蓝牙乱码误触发命令

`MCal START / CANCEL` 并不是磁力计或 I2C 自己异常触发，而是命令解析器收到了随机的 `M` 字节。

因此反复出现该日志是 UART1 RX 乱码的直接证据。

### 4.3 1.7V / 2.4V 虚电压是弱反灌/半供电现象

断开正常供电后仍出现 1.7V 到 2.4V 左右的电压，大概率来自：

- MCU UART TX 空闲高电平通过外设 IO 保护二极管反灌；
- CH340 模块 USB 供电或 TXD 高电平反灌；
- 蓝牙模块板载 LDO、LED、ESD 保护结构形成路径；
- I2C 上拉或其他模块电源轨耦合；
- 万用表高输入阻抗测到弱电压。

该电压通常不具备稳定带载能力，不能作为正常电源使用，但可能足以让蓝牙模块进入“不彻底断电、不正常工作”的半死机状态。

### 4.4 以前蓝牙偶发“不重启”也很可能是同一原因

以前断开蓝牙供电后仍有接近 2.4V 的虚电压维持模块，使蓝牙没有真正掉电，电源复位条件不满足。

结果是：

```text
蓝牙模块没有完全关机
内部状态没有清空
重新供电后不是干净启动
表现为不广播、不连接、串口不响应
必须重启根路电源才恢复
```

这与本次测到的虚电压现象一致。

## 5. 为什么 CH340 不影响，而蓝牙会影响

CH340 模块通常由 USB 正常供电，TXD 空闲态稳定为高电平，因此开发板 UART RX 不会收到大量随机毛刺。

蓝牙模块在接错、未上电、半上电或被虚电压维持时，TXD/RXD 可能处于不确定状态，容易输出乱码或毛刺，因此会影响开发板。

## 6. 软件层面的经验教训

### 6.1 不要让蓝牙裸单字节命令触发危险动作

原先蓝牙和 UART0 都可以直接通过单字符触发命令，例如：

```text
M
f
h
y
x
```

这对调试方便，但抗乱码能力很差。乱码中随机出现 `M` 的概率并不低。

建议：

- 危险命令只允许 UART0 触发；
- 蓝牙命令增加前缀或帧格式；
- 例如蓝牙必须发送 `$M\n`、`$X\n`，而不是裸 `M`；
- 或者蓝牙只允许读取状态/调参，不允许触发校准、启停电机等危险动作。

### 6.2 保留蓝牙安全版逻辑

建议保留以下保护：

- 上电默认关闭蓝牙 UART1 RX 中断；
- 需要时通过 UART0 命令手动打开蓝牙；
- 关闭蓝牙时将 PA8/PA9 配置为输入高阻，减少 IO 反灌；
- `M` 磁力计自动校准只允许 UART0 触发。

### 6.3 加 RX 风暴保护

可以增加保护逻辑：

```text
若 20ms 或 100ms 内 UART1 RX 字节数/中断数超过阈值
        ↓
判定蓝牙 RX 风暴
        ↓
自动关闭 UART1 RX 中断
        ↓
通过 UART0 打印 BT FAULT
```

这样即使以后接线错误或蓝牙异常，也不会拖死主系统。

## 7. 硬件层面的改进建议

### 7.1 TX/RX 串联电阻

推荐蓝牙 UART 接线加入限流保护：

```text
PA8 TX  -> 1k~4.7k -> 蓝牙 RXD
蓝牙 TXD -> 1k~4.7k -> PA9 RX
GND      -> GND
VCC      -> 蓝牙模块要求的稳定电源
```

作用：

- 接反时限流；
- IO 反灌时限流；
- 插拔毛刺影响减小；
- 双方 IO 冲突时降低损坏风险。

### 7.2 蓝牙 VCC 加下拉/放电电阻

为了避免蓝牙断电后被虚电压维持，可以尝试：

```text
蓝牙 VCC -> 10kΩ -> GND
```

如果 10k 不够，可以试 4.7k。正常 5V 供电时额外电流约为：

```text
10kΩ：0.5mA
4.7kΩ：约 1.06mA
```

目标是让蓝牙断电后 VCC 能快速降到接近 0V，至少应明显低于 1V。

### 7.3 蓝牙掉电时信号线也应高阻或断开

最规范的做法是：

```text
断蓝牙 VCC 的同时，让 TX/RX 也断开或变成高阻
```

可选方案：

- 软件关闭 UART1 后把 PA8/PA9 改为输入；
- 使用三态缓冲器，如 74HC125 / 74LVC125；
- 使用电平转换/总线开关；
- 使用 MOS 管电源开关并配合信号隔离。

## 8. 当前代码状态建议

当前建议保持在安全方向：

- 保留灰度 ADC 采样和 UART0 打印；
- 保留蓝牙安全保护；
- 暂时不要让蓝牙重新直接触发所有单字节命令；
- 后续如需恢复蓝牙功能，应加入命令帧、防乱码和 RX 风暴保护。

## 9. 本次排查的关键经验

1. **硬件接反可以表现得像软件死锁。** 看到系统卡死、串口乱输出、I2C 异常时，不要只盯软件。
2. **UART 乱码会误触发单字符命令。** 调试命令方便，但抗干扰能力差。
3. **未上电模块的 IO 仍可能反向供电。** 断 VCC 不代表模块真的断电。
4. **1.7V / 2.4V 这类虚电压不可当成正常电源。** 但它足以维持模块半死不活。
5. **CH340 正常不代表蓝牙也正常。** 两种模块内部电源和 IO 结构不同。
6. **隔离问题的最快办法是先屏蔽外设。** UART0-only 安全版帮助确认 I2C、ADC、主循环都没坏。
7. **接线确认永远优先。** 最终主故障是蓝牙接反。

## 10. 后续 TODO

建议后续按优先级处理：

1. 确认蓝牙正确接线后，观察 `BTRX / BTIRQ` 是否稳定；
2. 加 TX/RX 串联电阻；
3. 试验蓝牙 VCC 到 GND 的 10k 放电电阻；
4. 软件中保留 `M` 命令仅 UART0 可触发；
5. 增加蓝牙命令帧格式，避免裸单字节命令；
6. 增加 UART1 RX 风暴保护；
7. 完成这些后，再继续灰度 ADC 数据采集、标定和 PID 差速控制。

## 11. 一句话总结

这次“电路奇葩现象”的本质是：

> 蓝牙模块接反导致 UART1 收到乱码，乱码误触发单字符命令；同时系统存在 IO 反灌造成的虚电压，使蓝牙可能处于半上电状态，解释了以前蓝牙偶发不重启和必须重启根路电源的现象。
